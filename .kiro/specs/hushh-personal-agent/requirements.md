# Requirements Document

## Introduction

The Hushh Personal Data Agent is a privacy-first web application that serves as a personal AI-powered "digital butler." Built entirely with free and open-source tools, the application provides two distinct AI personas — Kai (financial advisor) and Nav (creator/brand agent) — to help users manage personal finance, wellness, networking, and daily productivity. The app is a Next.js PWA targeting desktop and mobile browsers with an Apple-grade UI experience.

## Glossary

- **Agent**: The AI-powered conversational assistant that processes user queries and returns actionable responses
- **Kai**: The financial advisor persona of the Agent, specialized in investment advisory, portfolio tracking, and spending analysis
- **Nav**: The creator and brand persona of the Agent, specialized in lifestyle concierge, wellness, and network management
- **Persona**: A named configuration consisting of a system prompt, display identity, and behavioral rules that shape Agent responses
- **Vault**: The encrypted personal data store holding user financial records, contacts, preferences, and documents
- **Consent_Toggle**: A per-data-field switch that controls whether the Agent may access a specific piece of user data
- **MCP (Master Consent Protocol)**: The consent management system governing all data access with audit logging
- **Audit_Trail**: A tamper-evident, append-only log of all Agent actions, data accesses, and consent changes
- **Session**: An authenticated user interaction period from login to logout or timeout
- **Conversation**: A multi-turn dialogue thread between the user and a specific Persona
- **Escalation**: The process of flagging a query or action for human operator review
- **PWA (Progressive Web App)**: A web application that can be installed on devices and supports offline capabilities
- **WebAuthn**: A browser-native API for passwordless biometric authentication
- **Vault_Entry**: A single encrypted record within the Vault, conforming to a JSON schema
- **Onboarding_Wizard**: The guided multi-step setup flow presented to new users on first login
- **Adapter**: A pluggable integration module that connects the application to an external data source or service
- **Notification**: A proactive alert generated by the Agent or system to inform the user of important events or insights
- **User_Profile**: The collection of personal preferences and settings that customize the Agent's behavior for a specific user

## Requirements

### Requirement 1: User Registration and Authentication

**User Story:** As a new user, I want to sign up and log in using familiar OAuth providers or email/password, so that I can securely access my personal agent.

#### Acceptance Criteria

1. WHEN a user navigates to the login page, THE Auth_System SHALL display options for Google OAuth, GitHub OAuth, and email/password authentication
2. WHEN a user selects Google OAuth, THE Auth_System SHALL redirect the user to Google's OAuth consent screen and, upon successful authorization, create or retrieve the user account and establish a Session
3. WHEN a user selects GitHub OAuth, THE Auth_System SHALL redirect the user to GitHub's OAuth consent screen and, upon successful authorization, create or retrieve the user account and establish a Session
4. WHEN a user submits a valid email and password for registration, THE Auth_System SHALL create a new user account with a securely hashed password and establish a Session
5. WHEN a user submits valid email and password credentials for login, THE Auth_System SHALL verify the credentials and establish a Session
6. IF a user submits invalid credentials, THEN THE Auth_System SHALL display a descriptive error message without revealing whether the email or password was incorrect
7. WHEN a Session is established, THE Auth_System SHALL issue a secure, HTTP-only session token with a configurable expiration period
8. WHEN a session token expires or the user logs out, THE Auth_System SHALL invalidate the Session and redirect the user to the login page

### Requirement 2: Biometric and Two-Factor Authentication

**User Story:** As a security-conscious user, I want to use biometric login and two-factor authentication, so that my personal data remains protected even if my password is compromised.

#### Acceptance Criteria

1. WHEN a user enables WebAuthn in account settings, THE Auth_System SHALL register the user's biometric credential using the WebAuthn browser API
2. WHEN a user with a registered WebAuthn credential visits the login page, THE Auth_System SHALL offer biometric authentication as a login option
3. WHEN a user authenticates via WebAuthn, THE Auth_System SHALL verify the biometric credential and establish a Session without requiring a password
4. WHEN a user enables two-factor authentication, THE Auth_System SHALL generate a TOTP secret and display a QR code for authenticator app enrollment
5. WHILE two-factor authentication is enabled for a user, THE Auth_System SHALL require a valid TOTP code after successful primary authentication before establishing a Session
6. IF a user provides an invalid TOTP code three consecutive times, THEN THE Auth_System SHALL lock the account for 15 minutes and log the failed attempts to the Audit_Trail

### Requirement 3: Conversational AI Chat Interface

**User Story:** As a user, I want to interact with my personal agent through a chat interface with streaming responses, so that I get a natural, real-time conversational experience.

#### Acceptance Criteria

1. THE Chat_Interface SHALL display a message input area, a scrollable conversation history, and a Persona selector
2. WHEN a user submits a text message, THE Chat_Interface SHALL send the message to the Agent and display the response using token-by-token streaming
3. WHILE the Agent is generating a response, THE Chat_Interface SHALL display a typing indicator and disable the submit button
4. WHEN the Agent completes a response, THE Chat_Interface SHALL re-enable the submit button and scroll the conversation to the latest message
5. THE Chat_Interface SHALL preserve the full Conversation history for the current Session and restore it when the user returns to an existing Conversation
6. WHEN a user starts a new Conversation, THE Chat_Interface SHALL create a new conversation thread with a unique identifier and timestamp
7. THE Chat_Interface SHALL render markdown formatting, code blocks, and structured data (tables, lists) in Agent responses
8. IF the Agent fails to generate a response within 30 seconds, THEN THE Chat_Interface SHALL display a timeout error message and allow the user to retry

### Requirement 4: Voice Input

**User Story:** As a mobile user, I want to speak my queries instead of typing, so that I can interact with the agent hands-free.

#### Acceptance Criteria

1. THE Chat_Interface SHALL display a microphone button adjacent to the text input area
2. WHEN a user activates the microphone button, THE Voice_System SHALL begin capturing audio using the Web Speech API and display a visual recording indicator
3. WHILE the Voice_System is recording, THE Voice_System SHALL transcribe speech to text in real-time and display the interim transcription in the input area
4. WHEN the user deactivates the microphone or speech recognition detects a pause, THE Voice_System SHALL finalize the transcription and populate the text input with the result
5. IF the browser does not support the Web Speech API, THEN THE Voice_System SHALL hide the microphone button and display no error
6. IF speech recognition fails to produce a transcription, THEN THE Voice_System SHALL display a message indicating the speech was not recognized and allow the user to retry

### Requirement 5: Agent Persona System (Kai and Nav)

**User Story:** As a user, I want to switch between Kai (financial advisor) and Nav (lifestyle/brand agent), so that I get specialized assistance tailored to my current need.

#### Acceptance Criteria

1. THE Persona_System SHALL provide two pre-configured personas: Kai (financial advisor and investment agent) and Nav (creator, brand, and lifestyle agent)
2. THE Persona_System SHALL store each Persona as a configuration object containing a name, avatar, system prompt, behavioral rules, and allowed data scopes
3. WHEN a user selects a Persona from the Persona selector, THE Persona_System SHALL apply the corresponding system prompt and behavioral rules to all subsequent Agent responses in that Conversation
4. WHEN the user switches Persona mid-Conversation, THE Persona_System SHALL start a new Conversation thread with the selected Persona and preserve the previous Conversation in history
5. THE Kai Persona SHALL respond to financial queries including portfolio analysis, spending breakdowns, investment recommendations, and proactive financial insights using data from the Vault
6. THE Nav Persona SHALL respond to lifestyle queries including booking requests, wellness recommendations, contact management, and brand/creator advisory using data from the Vault
7. WHILE a Persona is active, THE Persona_System SHALL restrict Agent data access to only the data scopes defined in that Persona's configuration and permitted by the user's Consent_Toggles

### Requirement 6: Personal Financial Concierge (Kai)

**User Story:** As a user, I want Kai to analyze my financial data and provide proactive insights, so that I can make better financial decisions.

#### Acceptance Criteria

1. WHEN a user asks Kai about spending patterns, THE Agent SHALL retrieve permitted financial records from the Vault and generate a categorized spending analysis
2. WHEN a user asks Kai about portfolio performance, THE Agent SHALL retrieve permitted portfolio data from the Vault and present holdings, gains/losses, and allocation breakdown
3. WHEN a user asks Kai for investment recommendations, THE Agent SHALL generate recommendations based on the user's risk profile and portfolio data, with appropriate disclaimers that the advice is informational only
4. THE Agent SHALL use mock financial data seeded in the Vault for the MVP, structured to match realistic financial record schemas
5. IF the user has not granted consent for financial data access, THEN THE Agent SHALL inform the user that financial data access is required and provide a link to the Consent management page


### Requirement 7: Consent Management (MCP)

**User Story:** As a privacy-conscious user, I want granular control over which data fields the agent can access, so that I maintain full ownership of my personal information.

#### Acceptance Criteria

1. THE Consent_Manager SHALL display a settings page listing all data categories (financial records, contacts, calendar, wellness, preferences) with individual Consent_Toggles for each data field
2. WHEN a user enables a Consent_Toggle, THE Consent_Manager SHALL record the grant with a timestamp, the data field identifier, and the granting user in the Audit_Trail
3. WHEN a user disables a Consent_Toggle, THE Consent_Manager SHALL immediately revoke Agent access to that data field and record the revocation in the Audit_Trail
4. WHILE a Consent_Toggle is disabled for a data field, THE Agent SHALL not read, process, or reference that data field in any response
5. THE Consent_Manager SHALL support scoped consent, allowing the user to grant access to a data field for a specific Persona only (Kai-only, Nav-only, or both)
6. WHEN a user views the consent history page, THE Consent_Manager SHALL display a chronological list of all consent changes with timestamps, data field names, actions (grant/revoke), and the associated Persona scope
7. THE Consent_Manager SHALL default all Consent_Toggles to disabled for new user accounts, requiring explicit opt-in

### Requirement 8: Encrypted Personal Vault

**User Story:** As a user, I want my personal data stored securely with encryption, so that my sensitive information is protected at rest.

#### Acceptance Criteria

1. THE Vault SHALL store all user data (financial records, contacts, preferences, documents) as Vault_Entries encrypted at rest using AES-256 encryption
2. THE Vault SHALL enforce a JSON schema for each Vault_Entry type, validating data structure on write operations
3. WHEN a user or the Agent writes a new Vault_Entry, THE Vault SHALL validate the entry against the appropriate JSON schema, encrypt the data, and store it with a timestamp and entry type identifier
4. WHEN the Agent or a user interface requests a Vault_Entry, THE Vault SHALL verify the requesting Persona's data scope and the user's Consent_Toggles before decrypting and returning the data
5. IF a Vault read request fails consent or scope verification, THEN THE Vault SHALL deny the request, return an access-denied error, and log the denied access attempt in the Audit_Trail
6. THE Vault SHALL support CRUD operations (create, read, update, delete) for all Vault_Entry types through a consistent API
7. WHEN a user deletes a Vault_Entry, THE Vault SHALL permanently remove the encrypted data and log the deletion in the Audit_Trail

### Requirement 9: Vault Data Schema Parsing and Serialization

**User Story:** As a developer, I want Vault entries to be reliably parsed from and serialized to JSON, so that data integrity is maintained across read/write operations.

#### Acceptance Criteria

1. THE Vault_Serializer SHALL parse JSON strings into typed Vault_Entry objects according to the defined JSON schemas
2. THE Vault_Serializer SHALL serialize typed Vault_Entry objects into JSON strings conforming to the defined JSON schemas
3. THE Vault_Pretty_Printer SHALL format Vault_Entry objects into human-readable, indented JSON strings
4. FOR ALL valid Vault_Entry objects, parsing the serialized output and then serializing again SHALL produce an identical JSON string (round-trip property)
5. IF the Vault_Serializer receives a JSON string that does not conform to any defined schema, THEN THE Vault_Serializer SHALL return a descriptive validation error identifying the non-conforming fields

### Requirement 10: Lifestyle and Wellness Concierge (Nav)

**User Story:** As a user, I want Nav to help me with wellness recommendations and booking requests, so that I can manage my lifestyle efficiently.

#### Acceptance Criteria

1. WHEN a user asks Nav for wellness recommendations, THE Agent SHALL generate personalized suggestions based on permitted user preferences and wellness data from the Vault
2. WHEN a user requests Nav to make a booking (restaurant, appointment, service), THE Agent SHALL generate a structured booking request containing the venue/service type, date, time, and preferences, and present it for user confirmation
3. WHEN the user confirms a booking request, THE Agent SHALL store the confirmed booking as a Vault_Entry and create a corresponding calendar event if calendar consent is granted
4. IF the user has not granted consent for wellness or preference data, THEN THE Agent SHALL provide general recommendations without personalization and inform the user that personalized suggestions require consent

### Requirement 11: Network and Relationship Management

**User Story:** As a user, I want the agent to help me manage my contacts and remind me about important dates, so that I can maintain strong personal and professional relationships.

#### Acceptance Criteria

1. THE Contact_Manager SHALL store contacts as Vault_Entries with fields for name, relationship type, communication history, important dates, and notes
2. WHEN a user asks the Agent about a contact, THE Agent SHALL retrieve the permitted contact data from the Vault and provide a summary of the relationship, recent interactions, and upcoming important dates
3. WHEN an important date (birthday, anniversary, follow-up) for a contact is within 7 days, THE Reminder_System SHALL generate a proactive reminder notification for the user
4. WHEN a user asks the Agent to add or update a contact, THE Agent SHALL validate the input, create or update the corresponding Vault_Entry, and confirm the change to the user
5. THE Contact_Manager SHALL support searching contacts by name, relationship type, or tags
6. IF the user has not granted consent for contact data access, THEN THE Agent SHALL inform the user that contact management features require consent and provide a link to the Consent management page


### Requirement 12: Calendar Integration

**User Story:** As a user, I want the agent to read and manage my calendar events, so that I can stay organized and schedule efficiently.

#### Acceptance Criteria

1. THE Calendar_System SHALL store calendar events as Vault_Entries with fields for title, description, start time, end time, location, and recurrence rules
2. WHEN a user asks the Agent to create a calendar event, THE Agent SHALL parse the request into structured event fields, present a confirmation summary, and upon confirmation create the event as a Vault_Entry
3. WHEN a user asks the Agent about upcoming events, THE Agent SHALL retrieve permitted calendar entries from the Vault and present them in chronological order with relevant details
4. WHEN a user asks the Agent to reschedule or cancel an event, THE Agent SHALL update or delete the corresponding Vault_Entry and confirm the change
5. WHEN a user asks the Agent about availability for a given time range, THE Agent SHALL check existing calendar entries and report available time slots
6. IF the user has not granted consent for calendar data access, THEN THE Agent SHALL inform the user that calendar features require consent

### Requirement 13: Audit Trail and Action Logging

**User Story:** As a user, I want a complete log of all agent actions and data accesses, so that I can review what the agent has done on my behalf.

#### Acceptance Criteria

1. THE Audit_System SHALL log every Agent action including data reads, data writes, consent changes, authentication events, and escalation events with a timestamp, user identifier, action type, and affected resource
2. THE Audit_System SHALL store audit log entries in an append-only format that prevents modification or deletion of existing entries
3. WHEN a user navigates to the audit trail page, THE Audit_System SHALL display a paginated, chronological list of audit entries with filtering by action type, date range, and Persona
4. THE Audit_System SHALL include a cryptographic hash chain where each entry references the hash of the previous entry, enabling tamper detection
5. WHEN a tamper is detected in the hash chain, THE Audit_System SHALL flag the affected entries and display a warning to the user
6. THE Audit_System SHALL retain audit log entries for the lifetime of the user account

### Requirement 14: Human-in-the-Loop Escalation

**User Story:** As a user, I want complex or sensitive queries to be flagged for human review, so that I can get expert assistance when the AI is uncertain.

#### Acceptance Criteria

1. WHEN the Agent determines that a query exceeds its confidence threshold or involves a sensitive action (financial transactions, data deletion, account changes), THE Escalation_System SHALL flag the query for human review
2. WHEN a query is escalated, THE Escalation_System SHALL notify the user that the query has been forwarded for human review and provide an estimated response time
3. THE Escalation_System SHALL store escalated queries as Vault_Entries with the original query, Agent's preliminary response, escalation reason, and status (pending, in-review, resolved)
4. WHEN an operator resolves an escalated query, THE Escalation_System SHALL deliver the resolution to the user and update the escalation status
5. THE Admin_Dashboard SHALL display all pending and resolved escalations with filtering by status, date, and Persona
6. THE Admin_Dashboard SHALL require separate operator authentication with role-based access control

### Requirement 15: Progressive Web App Support

**User Story:** As a mobile user, I want to install the web app on my device and use it like a native app, so that I get a seamless experience on any device.

#### Acceptance Criteria

1. THE Application SHALL serve a valid web app manifest with app name, icons, theme color, and display mode set to standalone
2. THE Application SHALL register a service worker that caches static assets for offline shell loading
3. WHEN a user visits the Application in a supported browser, THE Application SHALL trigger the browser's install prompt when PWA installation criteria are met
4. WHILE the device is offline, THE Application SHALL display cached UI shell with a message indicating limited functionality until connectivity is restored
5. THE Application SHALL be responsive, rendering correctly on viewport widths from 320px to 2560px

### Requirement 16: Premium UI and Theming

**User Story:** As a user, I want a clean, premium-feeling interface with dark and light modes, so that the app feels polished and comfortable to use.

#### Acceptance Criteria

1. THE UI_System SHALL implement a design system using Tailwind CSS and shadcn/ui components with consistent spacing, typography, and color tokens
2. THE UI_System SHALL support light mode and dark mode themes, persisting the user's preference across Sessions
3. WHEN a user toggles the theme, THE UI_System SHALL apply the selected theme immediately without a page reload
4. THE UI_System SHALL render all interactive elements with appropriate focus indicators, ARIA labels, and keyboard navigation support
5. THE UI_System SHALL display smooth transitions and animations for page navigation, modal dialogs, and chat message appearance with a maximum animation duration of 300ms
6. WHILE content is loading, THE UI_System SHALL display skeleton placeholders matching the layout of the expected content

### Requirement 17: Mock Data Layer

**User Story:** As a developer, I want a mock data layer that simulates real financial, contact, and calendar data, so that the MVP can demonstrate full functionality without external API dependencies.

#### Acceptance Criteria

1. THE Mock_Data_System SHALL provide seed data for financial records (transactions, portfolio holdings, spending categories), contacts (10 sample contacts with relationship data), and calendar events (20 sample events across a 30-day range)
2. THE Mock_Data_System SHALL conform to the same JSON schemas used by the Vault for real data
3. WHEN the application starts in development mode, THE Mock_Data_System SHALL seed the database with mock data if no existing user data is present
4. THE Mock_Data_System SHALL be replaceable with real API integrations without changes to the Agent, Vault, or UI layers

### Requirement 18: LLM Integration and Configuration

**User Story:** As a developer, I want a configurable LLM integration layer, so that the application can work with Ollama locally or any compatible API endpoint.

#### Acceptance Criteria

1. THE LLM_Service SHALL support connecting to Ollama running locally and to any OpenAI-compatible API endpoint, configurable via environment variables
2. THE LLM_Service SHALL send requests containing the active Persona's system prompt, conversation history, and the user's latest message to the configured LLM endpoint
3. THE LLM_Service SHALL stream response tokens back to the Chat_Interface as they are generated
4. IF the LLM endpoint is unreachable, THEN THE LLM_Service SHALL return an error message to the Chat_Interface indicating the AI service is unavailable and log the connection failure
5. THE LLM_Service SHALL enforce a maximum context window size by truncating older conversation history when the token limit is approached, preserving the system prompt and most recent messages
6. THE LLM_Service SHALL sanitize all user input before including it in LLM requests to prevent prompt injection attacks

### Requirement 19: User Onboarding Flow

**User Story:** As a new user, I want a guided onboarding experience that introduces me to Kai and Nav and helps me set up my preferences, so that I can start using the agent effectively from day one.

#### Acceptance Criteria

1. WHEN a user completes registration and logs in for the first time, THE Onboarding_Wizard SHALL present a multi-step guided setup flow before navigating to the main application
2. THE Onboarding_Wizard SHALL include an introduction step that presents the Kai and Nav personas with visual previews of each persona's avatar, name, and capabilities
3. WHEN the user reaches the preferences step, THE Onboarding_Wizard SHALL collect initial financial preferences (risk tolerance level, investment goals) for Kai and lifestyle preferences (wellness interests, favorite service categories) for Nav
4. WHEN the user reaches the consent step, THE Onboarding_Wizard SHALL display Consent_Toggles for each data category and allow the user to opt in to data sharing before completing setup
5. THE Onboarding_Wizard SHALL display a skip button on each step, allowing the user to bypass onboarding and configure preferences later from settings
6. WHEN the user completes or skips the Onboarding_Wizard, THE Application SHALL store the onboarding completion status in the user's account so that the Onboarding_Wizard is not presented on subsequent logins
7. THE Onboarding_Wizard SHALL use iOS-style step transitions with smooth animations and a progress indicator showing the current step out of total steps

### Requirement 20: Conversation History Management

**User Story:** As a user, I want to view, search, and manage my past conversations, so that I can revisit previous advice and maintain an organized history.

#### Acceptance Criteria

1. THE Conversation_Manager SHALL display a sidebar or dedicated page listing all past Conversations with each entry showing the conversation title, associated Persona (Kai or Nav), and the timestamp of the last message
2. WHEN a user enters a search query in the conversation search field, THE Conversation_Manager SHALL filter the conversation list to show only Conversations containing messages that match the keyword
3. WHEN a user selects a past Conversation from the list, THE Conversation_Manager SHALL load the full conversation history into the Chat_Interface and allow the user to continue the dialogue
4. WHEN a user requests deletion of a Conversation, THE Conversation_Manager SHALL permanently remove the Conversation and all associated messages from storage and log the deletion in the Audit_Trail
5. THE Conversation_Manager SHALL support filtering the conversation list by Persona, allowing the user to view only Kai conversations, only Nav conversations, or all conversations
6. WHEN a new Conversation is created, THE Conversation_Manager SHALL generate a default title based on the first user message and allow the user to rename the Conversation

### Requirement 21: User Profile and Preferences

**User Story:** As a user, I want to manage my personal profile and preferences, so that the agent can provide personalized advice and recommendations.

#### Acceptance Criteria

1. THE Profile_System SHALL provide a profile page with editable fields for display name, avatar image, timezone, and preferred language
2. THE Profile_System SHALL provide a financial preferences section with fields for risk tolerance level (conservative, moderate, aggressive), investment goals, and income range, used by the Kai Persona to personalize financial advice
3. THE Profile_System SHALL provide a lifestyle preferences section with fields for wellness interests, dietary preferences, and favorite service categories, used by the Nav Persona to personalize lifestyle recommendations
4. WHEN a user updates any profile or preference field, THE Profile_System SHALL store the updated value as a Vault_Entry with the corresponding Consent_Toggle association and log the change in the Audit_Trail
5. WHILE the Agent is generating a response, THE Agent SHALL retrieve the user's permitted preferences from the User_Profile and incorporate relevant preferences into the response context
6. THE Profile_System SHALL be accessible from the application settings at any time, allowing the user to edit preferences without restrictions
7. IF the user has not granted consent for a specific preference category, THEN THE Agent SHALL not use preferences from that category when generating responses

### Requirement 22: Integration Adapter Architecture

**User Story:** As a developer, I want a pluggable integration adapter architecture, so that new data sources and external services can be added without modifying core application logic.

#### Acceptance Criteria

1. THE Adapter_System SHALL define a standard Adapter interface specifying methods for authentication, data retrieval, data transformation, and error handling that all integration modules must implement
2. THE Mock_Data_System SHALL implement the Adapter interface, serving seed data through the same contract used by real integrations
3. THE Adapter_System SHALL provide an Adapter registry that maps data source identifiers to Adapter implementations, allowing runtime selection of active Adapters via environment configuration
4. WHEN an Adapter encounters an authentication failure or data retrieval error, THE Adapter SHALL handle the error internally and return a structured error response without propagating implementation details to the Agent, Vault, or UI layers
5. WHEN a new Adapter is added to the system, THE Adapter_System SHALL require no modifications to the Agent, Vault, or UI layers, only registration in the Adapter registry and implementation of the standard Adapter interface
6. THE Adapter_System SHALL include a documentation template specifying the required methods, expected data formats, authentication patterns, and error handling conventions for creating new Adapters

### Requirement 23: Notification and Proactive Alert System

**User Story:** As a user, I want to receive proactive notifications and alerts from the agent, so that I stay informed about important events, financial changes, and relationship milestones without having to ask.

#### Acceptance Criteria

1. THE Notification_System SHALL provide an in-app notification center displaying a list of notifications organized by read and unread status with timestamps
2. THE Notification_System SHALL support the following notification types: financial alerts (spending threshold breaches, portfolio value changes), contact reminders (birthdays, follow-up dates), calendar reminders (upcoming events), and system notifications (security alerts, consent changes)
3. WHEN a user selects a Notification, THE Notification_System SHALL navigate the user to the relevant Conversation, Vault_Entry, or application page associated with that Notification
4. THE Notification_System SHALL provide a notification preferences page where the user can enable or disable notifications for each notification type independently
5. WHERE the user has opted in to browser push notifications, THE Notification_System SHALL deliver push notifications via the Web Push API for the PWA, respecting the user's per-type notification preferences
6. WHILE a Consent_Toggle is disabled for a data category, THE Notification_System SHALL suppress all notifications related to that data category, generating no alerts for data the user has not consented to share
7. WHEN a new Notification is generated, THE Notification_System SHALL display an unread badge count on the notification icon in the application header
